# render-bubble

Takes the enriched slide array produced by the pptx image pipeline (`parseFile` → `captionImages`) and renders it as a bubble map where **every slide is its own bubble** — sized by how much content it carries, packed edge-to-edge with every other slide's bubble in an organically clustered (not grid- or ring-shaped) arrangement. At rest every bubble is a plain, flat `var(--accent)` circle showing its heading, truncated to ~6 words and sized relative to that bubble's own diameter (see `.slide-label`) so it stays legible even on a small bubble — no index number, no legend, no per-theme colour. Clicking a bubble **grows it in place**, lightens its fill to a washed-out tint of the accent, and swaps that short label for its full heading plus a short body preview inside; any images on that slide bloom out as small thumbnail circles around the now-larger bubble. Every other slide bubble clears out of the way to make room, at full opacity — nothing dims.

**This replaced an earlier, more cluttered version — don't drift back toward it.** The previous iteration coloured every resting bubble a dark shade of the accent (vs. a light shade for children) and, on click, bloomed a *separate* small circle per body paragraph plus a thumbnail per image around it. In practice the tiny per-paragraph text circles read as illegible once the map had to shrink to fit a viewport (see the note below on why the whole map now scales to fit at rest). The fix was dropping that whole child level: no separate text-bubble children, ever — a slide's heading still lives on its own bubble at rest (just truncated and responsively sized, see `.slide-label`), and its fuller heading + a short body preview move inside that same bubble once it's the active one. If a future request asks for "the full body visible without clicking" or similar, the actual fix is more restraint (bigger active bubble, shorter preview), not reintroducing per-paragraph satellite circles.

**Nothing here is placed on a perfect grid, ring, or even spacing — everything is organically clustered.** Both the top-level slide packing and (when a slide is active) its image cluster are seeded with deterministic pseudo-random jitter (`seededRandom()`) and then relaxed into a non-overlapping arrangement, rather than laid out on a grid or a ring with circles spaced by exact angle increments. An earlier version seeded the top-level pack on a literal grid and placed every child at the exact same distance from its parent, evenly spaced by angle — both are collision-free, but they read as mechanical and "sequentially spread out" rather than like bubbles that found their own resting place. If a change needs a denser or looser cluster, adjust the relaxation constants (Step 3) — don't reach for a grid or an even-angle ring, that's the thing that was rejected.

**Every level of this map is a circle. There is no rectangle, card, or popup anywhere, ever — that's the whole point of a bubble map.** Two earlier versions of this renderer got this wrong in different ways: one made a satellite dot's *panel* a big rounded-rectangle card concentric with the dot; the next tried anchoring that same rectangle flush against the bubble's edge instead of centering it. Both still read as "a modal appeared," just with different amounts of anchoring — because the shape itself (a card) is what reads as a popup, not its position. The fix isn't a better-anchored rectangle, it's no rectangle: content lives inside the bubble itself (once active) or in another small circle (image thumbnails), never a card. If a future request asks for "the panel to feel more attached" or similar, the actual fix is to make sure it's a circle, not to keep tuning a rectangle's position.

**Themes are for organizing the input JSON only — they have no visual role at all.** An earlier version made themes the top-level packed bubbles, with slides orbiting them on a fixed-radius ring; that produced huge gaps between theme clusters and was rejected (see Step 3). A later version dropped the ring but still gave each theme its own colour and a legend to explain it; that was ALSO rejected — a legend for something that isn't otherwise interactive is dead weight. Themes still group slides in the JSON (Step 2) and still matter for authoring — a well-chosen theme grouping is a useful way to think through the deck — but nothing about a theme reaches the rendered page: no bubble, no colour, no legend.

**Colour marks state (resting vs. active), not hierarchy or theme identity.** Every bubble is `var(--accent)` at rest; the active bubble alone switches to a lightened `color-mix()` tint (see `.slide-bubble.active` in the CSS) so it's unmistakable which one is open, even before you read its content. There is no more dark-parent/light-child colour pairing — that belonged to the old per-paragraph child-bubble design (see the note above) and no longer applies now that a slide's own bubble holds its own content. Image thumbnails reuse the same active-tint colour so they read as part of the bubble that's currently open. Don't reintroduce hue variation per theme, or a second shade reserved for "child" bubbles — colour now has exactly one job (resting vs. active).

**Like render-gallery, this does not consume the generic `transcript-to-html` extraction schema.** That schema's `image_description` is text-only — the actual image never survives it. This format needs the real embedded image, so it must be given the raw enriched slide array directly: one object per slide, shaped `{ slide, heading, body[], images[] }`, where each entry in `images[]` is `{ base64, mime_type, caption }`.

**Unlike timeline/gallery, this renderer is data-driven, not markup-duplicated.** Instead of copy-pasting a `.node`/`.card` HTML block per item, you fill a single JSON blob describing themes → slides → images, and the template's JS computes every bubble's size, growth, and position at load/click time. This is deliberate — the packing (Step 3) depends on how many slides/images exist and how much content each carries, which isn't known until you've read the deck, so it has to run generically at runtime rather than being hand-placed per item. Don't hand-write `.slide-bubble` / `.image-dot` DOM yourself — the only thing you author is the JSON.

## Step 1: Read the enriched slide data

- Input is the array output by `captionImages(await parsePptx(...))`.
- Fields per slide: `slide` (number), `heading` (string), `body` (string array), `images` (array, empty if none).
- Read every slide before grouping into themes (Step 2) — don't theme-cluster from a partial read.
- Do not fabricate headings, body text, or captions beyond what's given. If `body` is empty, omit it rather than inventing content.
- **`body[]` is shown as a single short preview inside the active bubble** (the template joins it and truncates to `BODY_PREVIEW_MAX_CHARS`, see Step 3) — it is no longer split into separate child bubbles. Still keep entries reasonably short and substantive; a long run-on `body` just gets truncated with an ellipsis once the slide is active, losing whatever ran past the limit, so front-load the most important point of a slide's body rather than burying it.

## Step 2: Identify themes

The input has no theme field — you assign one. Read all slide headings and body text, then group slides into a small number of logical themes/sections:

- Aim for roughly 3–8 themes for a typical 10–40 slide deck — fewer if the deck is short, more only if it's genuinely sprawling. Avoid two failure modes: one giant theme containing everything (defeats the point of clustering) and a theme per slide (same problem, no clustering happened).
- Every slide belongs to exactly one theme.
- If the deck has explicit section-divider slides (a slide that's just a big title with little/no body), treat those as a strong signal for where one theme ends and the next begins.
- Theme names: short (2–4 words), drawn from the actual content — e.g. "Market Landscape", "Product Roadmap" — never generic filler like "Section 1".
- Preserve original slide order within each theme; preserve theme order as first-encountered in the deck.

## Step 3: The packing

The template's JS (`layoutBubbles()` in the `<script>` block) does all positioning generically from the data you provide — you never compute a pixel position yourself. This section documents how it works so you know why bubbles pack tightly instead of scattering with gaps, why a bubble grows on click instead of opening a panel, and where the tunable constants live if a particular deck needs a different density.

**`seededRandom(seed)`, used throughout Levels 1 and 2:** a small deterministic hash (mulberry32-style) producing a pseudo-random float in `[0,1)` from an integer seed. Every "random" placement in this template uses this, never `Math.random()` — the script reruns from scratch on every page load, so a real-random seed would reshuffle the whole map (and any active slide's image cluster) on every reload, which reads as glitchy rather than alive. A seeded hash gives an organic-looking arrangement that's still the *same* arrangement every time the same data renders.

**Level 1 — slide bubbles, packed via `packCircles()` (a force-relaxation packer seeded from random jitter, not a grid or a ring):**
- Diameter is area-proportional to a content weight (`1 + bodyCharCount/220 + imageCount*1.4`), scaled between `SLIDE_MIN_D` and `SLIDE_MAX_D` via `sqrt(weight / maxWeight)` — a slide with ~4x the content of another gets ~2x the diameter (matching area, not linear diameter), the standard bubble-chart convention. `SLIDE_MIN_D` has real headroom (not just visual variety) since every resting bubble shows its truncated heading (`.slide-label`, font-size `clamp()`'d relative to `--d`) — too small a floor here and the smallest bubbles' labels stop being legible.
- All slide bubbles across every theme are packed together as one flat set — **themes do not get their own bubble or ring; they only group slides in the JSON.** See the note at the top of this file about why an earlier ring-per-theme version was rejected.
- The packer seeds every circle at a random angle and radius from the origin (`seededRandom`, spread scaled to roughly fit the expected cluster size), **not a grid** — a grid seed still converges to a valid non-overlapping pack after relaxation, but visibly kept its row/column structure (the centroid-pull is uniform and gentle, so it closes gaps without actually reshuffling neighbours), which read as "sequentially spread out" rather than a native, organically clustered blob. Random seeding removes the underlying structure the relaxation would otherwise preserve.
- After seeding, it runs `PACK_ITERATIONS` rounds of: pull every circle a little toward the shared centroid (closes gaps), then resolve any pairwise overlap by pushing the two circles apart along the line between their centers (keeps them from overlapping). This is the same family of technique bubble-chart libraries use — a small physics simulation — and converges to an organic, edge-to-edge packed cluster.
- The centroid-pull strength decays linearly to zero over the run (`pull = 0.02 * (1 - iter/PACK_ITERATIONS)`). A constant-strength pull left a small chronic overlap even after hundreds of iterations — decaying it to zero lets collision resolution have the final say, so the packing actually settles.
- The canvas is sized to the bounding box of the packed cluster plus `CANVAS_MARGIN` on every side, then every circle is shifted into positive coordinates — there's no arbitrary fixed canvas size, so there's never leftover dead space around the cluster. `CANVAS_MARGIN` is generous enough to fit the largest possible *active* bubble plus its image bloom (see below), not just a resting one.

**The active bubble grows in place rather than opening beside itself — this is the whole interaction model, there is no separate "expanded panel."** Clicking a slide bubble computes `expandedD = clamp(slide.d * ACTIVE_SCALE, ACTIVE_MIN_D, ACTIVE_MAX_D)` and animates that bubble's own `width`/`height` (via CSS transition on `.slide-bubble`, which — unlike the transition list on an ordinary `.bubble` — keeps width/height on the calm `--ease` curve rather than dropping them) up to that size, swaps in its lightened active fill, and fills it with a heading + `truncate(body.join(' '), BODY_PREVIEW_MAX_CHARS)`. `ACTIVE_MIN_D`/`ACTIVE_MAX_D` bound this so a tiny resting bubble still grows roomy enough to hold text legibly, and a huge one doesn't swallow the map.

**Image thumbnails, clustered via `packChildrenAroundParent()`, built fresh at click time (not at initial layout) around whichever slide is active:**
- Only images bloom around the active bubble now — there is no separate per-paragraph text child anymore (see the note near the top of this file for why that was dropped). If a slide has no images, nothing blooms; its content is entirely inside the grown bubble itself.
- Built at click time, not upfront, because the cluster's radius depends on `expandedD` — the size the bubble just grew to — which isn't known until the click happens. `selectSlide()` builds a fresh `.childcluster` container, packs the image thumbnails into it, and removes it again on `collapseAll()` (rebuilt from scratch next time, rather than kept around `hidden`).
- Positioning itself works exactly like Level 1's packer in miniature: seed each thumbnail at a random angle and near-minimum distance from the parent (`seededRandom`, seeded per-slide so the same slide always blooms the same way), then relax — pull weakly toward the parent's center, re-clamp outside its surface if that pull moved it inside, resolve pairwise overlaps between thumbnails. An earlier version placed every child on one perfectly even ring (same radius, angles spaced by exactly `2π/n`) — mathematically collision-free but mechanical-looking; don't reintroduce that.
- Thumbnail diameter is `expandedD * IMAGE_FRAC`, clamped to `[IMAGE_MIN_D, IMAGE_MAX_D]` so thumbnails stay legible/clickable regardless of the parent's size. Clicking one **zooms it in place** via `transform: scale(var(--zoom))`, where `--zoom` is computed per-dot as `IMAGE_EXPANDED_D / thatDot'sOwnDiameter` — cheap, no reflow, safe to animate since an image (unlike wrapped text) has no internal layout to fight while it grows. Its caption appears as a small text label positioned below the zoomed circle (a sibling element, not inside the circle) — never inside a card.
- After clustering, `slide.footprintR` is set to the actual maximum distance from the parent's center to any thumbnail's outer edge (not an analytic formula), falling back to just `expandedD/2` if there are no images — this is what `relaxNeighbors()` (Step 4) uses to clear space, so it stays accurate no matter how the organic clustering happens to settle for that slide.

**The map fits the viewport at rest, but zooms to full (natural) scale on an expanded slide — these are two different `#canvas` transform scales, not one.** `layoutBubbles()` computes `restScale` (shrinks the whole map via `transform: scale()` so every bubble is visible without scrolling on load, floored at `FIT_MIN_SCALE` so it can't shrink text into illegibility on a big deck) and applies it via `zoomToOverview()`. `selectSlide()` switches to `zoomToSlide()` instead — full scale (1), centered on that one slide — because the active bubble's heading/body/image sizing all assume natural scale; leaving `restScale` applied while a bubble is expanded shrinks its text along with the rest of the map, which is what "readable at rest, illegible once you actually click a bubble" looked like the one time this was tried. Collapsing (`collapseAll` + `zoomToOverview()`) zooms back out. Don't collapse these into a single fixed scale to "simplify" it — the whole point is that the two states want different scales.

**Tunable constants** (top of the `<script>` block — change here if a specific deck needs different density, don't hardcode alternate values inline elsewhere):
```js
SLIDE_MIN_D = 110, SLIDE_MAX_D = 280, SLIDE_GAP = 14 // resting size — SLIDE_MIN_D must stay big enough for .slide-label to read cleanly
ACTIVE_SCALE = 1.8, ACTIVE_MIN_D = 210, ACTIVE_MAX_D = 380 // how big a bubble grows when it's the active one
BODY_PREVIEW_MAX_CHARS = 130 // body[] is joined + truncated to this before showing inside the active bubble
IMAGE_MIN_D = 38, IMAGE_MAX_D = 92, IMAGE_FRAC = 0.28, IMAGE_EXPANDED_D = 220, CHILD_GAP = 12, CHILD_PARENT_GAP = 20
PACK_ITERATIONS = 400
RELAX_ITERATIONS = 200 // see relaxNeighbors() in Step 4 — the "clear space around the active bubble" pass
FIT_MIN_SCALE = 0.4 // floor for the whole-map fit-to-viewport zoom (see the note above)
```

## Step 4: Click / expand behaviour

- **Collapsed state:** every slide is a plain `.slide-bubble` — flat `var(--accent)` fill, no border, showing its heading truncated to ~6 words via `.slide-label` (a native `title` attribute also carries the untruncated heading as a hover tooltip). There is no legend (see the colour note near the top of this file for why) and no per-slide index number shown.
- **Click a slide bubble → it grows in place, lightens, and reveals its content; its neighbors clear out of the way.** There's no separate panel or child-cluster-of-text anymore (see the notes above) — the bubble itself is the content surface. Any images bloom around it (built fresh, see Step 3). The clicked bubble itself never moves — it's the fixed anchor — but `relaxNeighbors()` nudges every *other* slide bubble outside the newly-active bubble's `footprintR` (its own grown radius plus however far its image bloom extends), so nothing bloomed ever lands on top of a neighbor.
- **Neighbors clear space but do NOT dim.** An earlier version added a `dimmed` (25% opacity) class to every other bubble while one was active — dropped along with the per-paragraph child-bubble redesign, since the reference look this renderer now matches keeps every resting bubble at full, equal visual weight regardless of which one is active. Don't reintroduce a dimmed state without being asked; if contrast between active/inactive is ever an issue again, that's what the active bubble's size + lightened fill are already for.
- **Motion is springy for position, calm for size.** `.slide-bubble`'s `left`/`top` transition uses `--spring-ease` (`cubic-bezier(.34,1.56,.64,1)`, a slight overshoot) — bubbles nudging each other aside is supposed to read as organic and slightly wobbly. Its `width`/`height`/`background` transition uses the calm `--ease` instead — growing into the active state is a deliberate camera-like zoom, not bubble-on-bubble physics, so mixing in the springy curve there would read as an inconsistent wobble on the one bubble that isn't actually being displaced by anything. Collapsing (`resetPositions()`) animates every bubble back to its packed resting position and size with the same pairing. Don't reuse `--spring-ease` for anything that isn't a bubble physically displacing another bubble.
- **Image children** (`.image-dot`, one per image) are small round thumbnails, visible as soon as the parent's cluster blooms. **Click one → it zooms in place** (`transform: scale()`, see Step 3) and a caption label fades in below it (a sibling text element, not inside any card). Click it again, or click elsewhere, to shrink it back. This zoom keeps the calm `--ease`, not `--spring-ease` — it's the image growing in its own reserved space, not one bubble displacing another.
- **Collapsing:** click the slide bubble again (toggles closed), click an expanded image dot again, or click empty canvas background — collapses whatever's open, shrinks the bubble back to its resting size, clears its heading/body text and image cluster, and springs every displaced neighbor back to its resting position. A click anywhere inside an open `.childcluster` (an image thumbnail, the gap between them) does not bubble up and collapse the slide. Max one slide expanded at a time; expanding a new one auto-collapses whichever was open before relaxing space around the new one.

## Step 5: Fill the template

Use `assets/bubble-template.html`. Copy it to your working directory.

1. Fill `{{BUBBLE_TITLE}}`, `{{BUBBLE_SUBTITLE}}`, `{{AUTHOR}}`.
2. Build the theme/slide/image structure from Steps 1–2 and replace the entire `{{BUBBLE_DATA}}` token (inside the `<script type="application/json" id="bubble-data">` tag) with one JSON object — no markdown fencing, no surrounding commentary:
   ```json
   {
     "themes": [
       {
         "name": "Theme Name",
         "slides": [
           {
             "index": 3,
             "heading": "...",
             "body": ["...", "..."],
             "images": [{ "src": "__IMAGE_SLIDE_3_1__", "caption": "..." }]
           }
         ]
       }
     ]
   }
   ```
3. **Image placeholders — critical, same rule as render-gallery:** never inline actual base64 data yourself. For every image on every slide, set its `src` to the exact literal token `__IMAGE_SLIDE_<slide>_<n>__` (`<slide>` = that slide's number, `<n>` = 1-based index of the image within that slide's `images[]`, so a slide with 2 images gets `__IMAGE_SLIDE_5_1__` and `__IMAGE_SLIDE_5_2__`). A code step after your output does a plain string substitution of each token for the real `data:` URI. Don't invent a filename or alter the token in any way.
4. `body` may still be an array of multiple strings — keep them as separate short entries as authored, even though the template now joins and truncates them into one preview when a slide is active (see Step 1/3). Don't pre-join them yourself into the JSON; hand over the array as given.

## Step 6: Chrome

Copy the toolbar HTML/CSS/JS verbatim from the existing templates — do not rewrite it. 5-colour theme picker (blue, green, purple, orange, **red** — same palette as render-gallery, blue is the default). Dark mode toggle. Download PDF button targets `#canvas` (the full bubble map at its actual computed size), not `#canvas-wrap` (the scrollable viewport) or the whole page — targeting the wrapper would only capture whatever's currently scrolled into view.

**`.doc-header` (the title block) has a `::before` gradient scrim behind it — don't remove it or turn it into a solid card.** Unlike every other renderer's header, this one sits directly on top of a canvas that scrolls and packs bubbles edge-to-edge underneath it, so without something behind the text, whatever bubble happens to be scrolled under the title collides with it and both become hard to read — that's a real bug that was reported ("looks tacky"). The fix is a soft radial gradient in `var(--bg)` (the page's own background colour), strongest immediately behind the text and fading to fully transparent within its own bounds — deliberately NOT a flat, hard-edged rectangle, which would read as its own competing card fighting the bubbles for attention rather than the bubble map staying the main focus. If a future change wants the header more/less legible, adjust the gradient's stops/opacity, don't swap it for a solid background.

Scrollbars are hidden everywhere on every renderer (the global `* { scrollbar-width: none; ... } *::-webkit-scrollbar { display: none; }` rule right after the box-sizing reset) — scrolling still works, there's just no visible track/thumb. The canvas also supports click-and-drag panning since a big bubble map with invisible scrollbars needs an obvious way to explore it — don't remove that.

**`#canvas-wrap` must stay plain block layout — never `display: flex` + `align-items`/`justify-content: center`.** That combination on a scrollable (`overflow: auto`) container is a real trap: once `#canvas` is bigger than the wrap, flex's default ("unsafe") centering lets the child overflow equally on both sides, but scroll offsets can't go negative, so the overflow on the leading side (roughly the top-left of the map) becomes permanently clipped and unreachable by scrolling or dragging. This was a real bug: bubbles rendered visibly on screen but couldn't be scrolled to. Plain block layout has no such trap — `#canvas` sits at `(0,0)` inside the wrap and every pixel is reachable across the full scroll range. The "starts centered on load" look, and the "whole map fits without scrolling" look (Step 3), come entirely from `layoutBubbles()`/`zoomToOverview()` setting `transform: scale()` plus `scrollLeft`/`scrollTop` after layout — that's sufficient on its own; don't reach for flex centering to get it "for free."

**Click-and-drag panning must be able to start from ANYWHERE, including on top of a bubble — not just on bare canvas background.** An earlier version's `mousedown` handler bailed out immediately if the target was inside a `.bubble`, on the theory that bubbles need that mousedown for their own click-to-select. That worked when bubbles were small with visible gaps between them, but once bubbles got bigger and packed edge-to-edge (Step 3), there was often no bare background left to grab — nearly every mousedown landed on a bubble, panning couldn't start, and dragging across the map just fired each bubble's ordinary click in turn instead of panning through the space. The fix (see the template's `enableDragPan()`): always start tracking on `mousedown` regardless of target, then decide after the fact — via a `DRAG_THRESHOLD` (6px) of pointer movement — whether this was a drag (pan) or a click (select). If it was a drag, a `document`-level capture-phase `click` listener swallows the click event that would otherwise still fire on mouseup, so whatever the pointer started or ended on top of doesn't also react as if it'd been clicked. Don't reintroduce a target-based bail-out on `mousedown` — always track, decide by movement distance instead.

## Step 7: Validate before delivering

1. Grep for any remaining `{{` strings — anything left is a bug.
2. Confirm `{{BUBBLE_DATA}}` was replaced with valid JSON (e.g. pipe it through `node -e "JSON.parse(require('fs').readFileSync(0,'utf-8'))"`) — a syntax error here means the whole map fails to render.
3. Confirm no slide `heading`, `body` entry, or image `caption` contains the literal substring `</script` — that would terminate the data blob's script tag early and break the page. (Content this specific should never occur from a VLM caption or slide text, but check.)
4. Confirm every image across every slide has a correctly-numbered `__IMAGE_SLIDE_<slide>_<n>__` token and that no two images share a token.
5. Run `node --check` on the inline `<script>` block.

## Notes on extending

- If a third level of drill-down is ever needed (e.g. sub-bubbles on an image), follow the same pattern as the image cluster: content-sized circles, `packChildrenAroundParent()`-style random-seed-then-relax clustering around whichever single node is active, built fresh at click time. Don't introduce a rectangle at any level, don't lay children out on an even-angle ring, and don't introduce a different interaction model for one level.
- `packCircles()`, `packChildrenAroundParent()`, `seededRandom()`, and the tunable constants are the only things that should change if bubble density or organic "wobbliness" needs adjusting for a particular deck — don't add one-off inline overrides elsewhere in the script.
- Don't reintroduce a theme-level bubble, a ring-orbit layout baked into the initial map, a perfectly even-angle ring for child bubbles, a grid seed for the top-level pack, a rectangular content panel at any level (concentric or edge-anchored), or a legend — see the notes near the top of this file for why each of those was tried and rejected.
- Don't reintroduce per-theme colour or hue variation, per-paragraph text child bubbles, or a `dimmed` state for inactive neighbors — all of these were tried in an earlier version and explicitly dropped (see the notes near the top of this file and Step 4) in favour of the current label-at-rest / grows-and-lightens-when-active model. The resting heading label (`.slide-label`) stays, though — don't remove it, it's a deliberate, current part of the design, not a leftover.
