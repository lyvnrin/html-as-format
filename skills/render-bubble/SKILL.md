# render-bubble

Takes the enriched slide array produced by the pptx image pipeline (`parseFile` → `captionImages`) and renders it as a bubble map where **every slide is its own parent bubble** — sized by how much content it carries, packed edge-to-edge with every other slide's bubble in an organically clustered (not grid- or ring-shaped) arrangement — filled a dark shade of the currently selected accent colour. Clicking a slide's bubble reveals its **content as child bubbles**: one small circle per body paragraph and one thumbnail circle per image, each visibly smaller than its own parent and filled a light shade of the same accent, nestled organically around the now-active parent, while every other slide bubble clears out of the way to make room.

**Nothing here is placed on a perfect grid, ring, or even spacing — everything is organically clustered.** Both the top-level slide packing and each slide's child cluster are seeded with deterministic pseudo-random jitter (`seededRandom()`) and then relaxed into a non-overlapping arrangement, rather than laid out on a grid or a ring with circles spaced by exact angle increments. An earlier version seeded the top-level pack on a literal grid and placed every child at the exact same distance from its parent, evenly spaced by angle — both are collision-free, but they read as mechanical and "sequentially spread out" rather than like bubbles that found their own resting place. If a change needs a denser or looser cluster, adjust the relaxation constants (Step 3) — don't reach for a grid or an even-angle ring, that's the thing that was rejected.

**Every level of this map is a circle. There is no rectangle, card, or popup anywhere, ever — that's the whole point of a bubble map.** Two earlier versions of this renderer got this wrong in different ways: one made a satellite dot's *panel* a big rounded-rectangle card concentric with the dot; the next tried anchoring that same rectangle flush against the bubble's edge instead of centering it. Both still read as "a modal appeared," just with different amounts of anchoring — because the shape itself (a card) is what reads as a popup, not its position. The fix isn't a better-anchored rectangle, it's no rectangle: content is itself a small bubble, sized to its own content, sitting in the cluster around its parent. If a future request asks for "the panel to feel more attached" or similar, the actual fix is to make sure it's a circle, not to keep tuning a rectangle's position.

**Themes are for organizing the input JSON only — they have no visual role at all.** An earlier version made themes the top-level packed bubbles, with slides orbiting them on a fixed-radius ring; that produced huge gaps between theme clusters and was rejected (see Step 3). A later version dropped the ring but still gave each theme its own colour and a legend to explain it; that was ALSO rejected — a legend for something that isn't otherwise interactive is dead weight, and colour is put to better use signalling something the user actually needs at a glance (see the next note). Themes still group slides in the JSON (Step 2) and still matter for authoring — a well-chosen theme grouping is a useful way to think through the deck — but nothing about a theme reaches the rendered page: no bubble, no colour, no legend.

**Colour is one hue, two shades: dark for every parent, light for every child — never a theme identity.** `PARENT_CAT`/`CHILD_CAT` (see Step 3) are both `color-mix()` formulas referencing `var(--accent)`, one mixed toward black, one toward white. Two earlier versions got this wrong in different ways: one used 8 fixed, unrelated hues (blue, green, orange, purple...) regardless of which accent was selected in the chrome's theme picker, so picking "blue" still showed green and orange bubbles; the next fixed the hue but still varied the SHADE per theme (needing the legend the previous note describes) instead of using shade to mark parent-vs-child depth. The current design reserves the entire shade axis for depth: every slide bubble is the same dark shade, every child bubble (text or image) is the same light shade, regardless of which theme either belongs to. Don't reintroduce hue variation or per-theme shading — if a future request wants themes visually distinguishable again, that needs a genuinely different visual channel (e.g. an icon or a border pattern), not colour, because colour is already spoken for.

**Like render-magazine, this does not consume the generic `transcript-to-html` extraction schema.** That schema's `image_description` is text-only — the actual image never survives it. This format needs the real embedded image, so it must be given the raw enriched slide array directly: one object per slide, shaped `{ slide, heading, body[], images[] }`, where each entry in `images[]` is `{ base64, mime_type, caption }`.

**Unlike timeline/magazine, this renderer is data-driven, not markup-duplicated.** Instead of copy-pasting a `.node`/`.card` HTML block per item, you fill a single JSON blob describing themes → slides → images, and the template's JS computes every bubble's size and position at load time. This is deliberate — the packing (Step 3) depends on how many slides/images exist and how much content each carries, which isn't known until you've read the deck, so it has to run generically at runtime rather than being hand-placed per item. Don't hand-write `.slide-bubble` / `.text-bubble` / `.image-dot` DOM yourself — the only thing you author is the JSON.

## Step 1: Read the enriched slide data

- Input is the array output by `captionImages(await parsePptx(...))`.
- Fields per slide: `slide` (number), `heading` (string), `body` (string array), `images` (array, empty if none).
- Read every slide before grouping into themes (Step 2) — don't theme-cluster from a partial read.
- Do not fabricate headings, body text, or captions beyond what's given. If `body` is empty, omit it rather than inventing content.
- **Each `body[]` entry becomes its own small circular bubble** (see Step 3/4) — favour several short entries over one long run-on paragraph. Aim for **roughly 60–120 characters per entry** (about one short sentence or clause) — the template's child-bubble sizing (Step 3) comfortably fits entries in that range without scrolling; entries pushing past ~160 characters are the case that occasionally forces the rare-case scroll fallback. A `body` array of 3-4 short entries reads far better as bubbles than one 60-word paragraph crammed into a single circle. Don't invent breaks that change meaning, but if the original slide text has natural sentence, clause, or bullet boundaries, split on those rather than joining everything into one long string.

## Step 2: Identify themes

The input has no theme field — you assign one. Read all slide headings and body text, then group slides into a small number of logical themes/sections:

- Aim for roughly 3–8 themes for a typical 10–40 slide deck — fewer if the deck is short, more only if it's genuinely sprawling. Avoid two failure modes: one giant theme containing everything (defeats the point of clustering) and a theme per slide (same problem, no clustering happened).
- Every slide belongs to exactly one theme.
- If the deck has explicit section-divider slides (a slide that's just a big title with little/no body), treat those as a strong signal for where one theme ends and the next begins.
- Theme names: short (2–4 words), drawn from the actual content — e.g. "Market Landscape", "Product Roadmap" — never generic filler like "Section 1".
- Preserve original slide order within each theme; preserve theme order as first-encountered in the deck.

## Step 3: The packing

The template's JS (`layoutBubbles()` in the `<script>` block) does all positioning generically from the data you provide — you never compute a pixel position yourself. This section documents how it works so you know why bubbles pack tightly instead of scattering with gaps, and where the tunable constants live if a particular deck needs a different density.

**`seededRandom(seed)`, used throughout Levels 1 and 2:** a small deterministic hash (mulberry32-style) producing a pseudo-random float in `[0,1)` from an integer seed. Every "random" placement in this template uses this, never `Math.random()` — the script reruns from scratch on every page load, so a real-random seed would reshuffle the whole map (and every slide's child cluster) on every reload, which reads as glitchy rather than alive. A seeded hash gives an organic-looking arrangement that's still the *same* arrangement every time the same data renders.

**Level 1 — slide bubbles, packed via `packCircles()` (a force-relaxation packer seeded from random jitter, not a grid or a ring):**
- Diameter is area-proportional to a content weight (`1 + bodyCharCount/220 + imageCount*1.4`), scaled between `SLIDE_MIN_D` and `SLIDE_MAX_D` via `sqrt(weight / maxWeight)` — a slide with ~4x the content of another gets ~2x the diameter (matching area, not linear diameter), the standard bubble-chart convention.
- All slide bubbles across every theme are packed together as one flat set — **themes do not get their own bubble or ring; they only pick each slide's colour.** See the note at the top of this file about why an earlier ring-per-theme version was rejected.
- The packer seeds every circle at a random angle and radius from the origin (`seededRandom`, spread scaled to roughly fit the expected cluster size), **not a grid** — a grid seed still converges to a valid non-overlapping pack after relaxation, but visibly kept its row/column structure (the centroid-pull is uniform and gentle, so it closes gaps without actually reshuffling neighbours), which read as "sequentially spread out" rather than a native, organically clustered blob. Random seeding removes the underlying structure the relaxation would otherwise preserve.
- After seeding, it runs `PACK_ITERATIONS` rounds of: pull every circle a little toward the shared centroid (closes gaps), then resolve any pairwise overlap by pushing the two circles apart along the line between their centers (keeps them from overlapping). This is the same family of technique bubble-chart libraries use — a small physics simulation — and converges to an organic, edge-to-edge packed cluster. It's simpler to hand-roll than true front-chain circle packing and accurate enough for a layout computed once at load.
- The centroid-pull strength decays linearly to zero over the run (`pull = 0.02 * (1 - iter/PACK_ITERATIONS)`). A constant-strength pull left a small chronic overlap even after hundreds of iterations — collision resolution only gets one sequential sweep per round, so a still-active pull kept re-introducing tiny overlaps faster than the sweep could clear them. Decaying it to zero lets collision resolution have the final say, so the packing actually settles.
- The canvas is sized to the bounding box of the packed cluster plus `CANVAS_MARGIN` on every side, then every circle is shifted into positive coordinates — there's no arbitrary fixed canvas size, so there's never leftover dead space around the cluster.

**Level 2 — child bubbles (on an expanded slide), clustered via `packChildrenAroundParent()` around the ACTIVE parent only:**
- A slide's children are every body paragraph (as a `.text-bubble`) plus every image (as an `.image-dot` thumbnail), combined into **one set** and clustered around that slide's fixed center. An earlier version placed every child on one perfectly even ring (same radius, angles spaced by exactly `2π/n`) — mathematically collision-free, but mechanical-looking, the same "sequentially spread out" problem as the old grid seeding at Level 1. `packChildrenAroundParent()` instead seeds each child at a random angle and a near-minimum distance from the parent (`seededRandom`, seeded per-slide so the same slide always blooms the same way), then relaxes: pull every child weakly toward the parent's center, re-clamp it outside the parent's surface if that pull moved it inside, and resolve pairwise overlaps between children. Children end up nestling at varying distances and angles — some hugging the parent directly, others nested a little further out behind a sibling — the way real bubbles cluster against a larger one.
- This clustering only runs around whichever **one** slide is currently expanded, transiently, computed fresh on that click — it isn't baked into the initial layout of every slide, so it can't push the rest of the map apart the way the old theme-ring did (see the note near the top of this file).
- **`.text-bubble` diameter is content-based first, then capped to a fraction of its own parent — getting this required TWO changes together, not one.** It's `min(contentD, d * TEXT_MAX_PARENT_FRAC)`, where `contentD = TEXT_MIN_D + (TEXT_MAX_D - TEXT_MIN_D) * sqrt(charCount / longestParagraphInThisSlide)` (the same sqrt-of-length sizing every bubble here uses) and `d` is that specific parent's own diameter. Three versions of this got tried: (1) sizing purely from `contentD` with no cap — children could reach or exceed their own parent's size, destroying the hierarchy; (2) sizing purely as a fraction of the parent (0.22–0.5 of `d`, no content-based component) — guaranteed "smaller than parent" but made children far too small to hold real paragraph text, forcing a visible scrollbar on nearly every child; (3) the current version, content-based-then-capped, plus **raising `SLIDE_MIN_D`/`SLIDE_MAX_D` themselves** — capping alone, without also giving parents more absolute headroom, still weren't big enough for the cap fraction to clear the diameter a ~150-character paragraph actually needs. If children are still scrolling for typical content, check both: is `body[]` too long per entry (Step 1), and are parents big enough that `TEXT_MAX_PARENT_FRAC * d` clears roughly 180–200px for the deck's higher-weight slides? `overflow-y: auto` inside the circle remains a fallback for an unusually long paragraph, not the norm — and it must stay invisible (no `scrollbar-width`/`::-webkit-scrollbar` override on `.text-bubble`), inheriting the sitewide hidden-scrollbar rule like everything else; a visible scrollbar on a small circle reads as broken.
- `.image-dot` thumbnails are similarly relative — `d * IMAGE_FRAC`, clamped to `[IMAGE_MIN_D, IMAGE_MAX_D]` so they stay legible/clickable even for a small parent. Clicking one **zooms it in place** via `transform: scale(var(--zoom))`, where `--zoom` is computed per-dot as `IMAGE_EXPANDED_D / thatDot'sOwnDiameter` (not a shared constant, since dots are no longer all the same size) — cheap, no reflow, and safe to animate a dimension-like property here specifically because an image (unlike wrapped text) has no internal layout to fight while it grows. Its caption appears as a small text label positioned below the zoomed circle (a sibling element, not inside the circle) — never inside a card.
- After clustering, `slideEntry.footprintR` is set to the actual maximum distance from the parent's center to any child's outer edge (not an analytic formula) — this is what `relaxNeighbors()` (Step 4) uses to clear space for the bloom, so it stays accurate no matter how the organic clustering happens to settle for that slide.
- Pre-computed at layout time alongside everything else (not created on click) but kept `hidden` until that slide bubble expands — the cluster's positions are fixed from the start, expansion only reveals what was already computed.
- Only one slide is ever expanded at a time (see Step 4), so a child cluster never has to worry about colliding with another slide's bubble.

**Tunable constants** (top of the `<script>` block — change here if a specific deck needs different density, don't hardcode alternate values inline elsewhere):
```js
SLIDE_MIN_D = 110, SLIDE_MAX_D = 340, SLIDE_GAP = 12 // bigger than it looks like it needs to be — this is what gives children room
TEXT_MIN_D = 84, TEXT_MAX_D = 220, TEXT_MAX_PARENT_FRAC = 0.75, CHILD_GAP = 12 // content-based, capped to 75% of the parent's own d
IMAGE_MIN_D = 38, IMAGE_MAX_D = 92, IMAGE_FRAC = 0.28, IMAGE_EXPANDED_D = 220, CHILD_PARENT_GAP = 20
PACK_ITERATIONS = 400
RELAX_ITERATIONS = 200 // see relaxNeighbors() in Step 4 — the "clear space around the active bubble" pass
PARENT_CAT = shade(-0.3), CHILD_CAT = shade(0.42) // dark/light shades of var(--accent) — depth, not theme identity
```

## Step 4: Click / expand behaviour

- **Collapsed state:** every slide is a `.slide-bubble` showing its (possibly truncated to ~6 words) heading, filled `PARENT_CAT` (a dark shade of the accent, same for every slide regardless of theme) — this heading is the only text visible before a click, there is no separate "collapsed dot vs. expanded panel" split at the slide level, and there is no legend (see the colour note near the top of this file for why).
- **Click a slide bubble → its child bubbles bloom around it, and its neighbors clear out of the way.** `.childcluster` is a container that exists in the DOM the whole time (built and positioned at layout time, see Step 3) but sits `hidden` until this slide is selected. The clicked bubble itself never moves — it's the fixed anchor — but `relaxNeighbors()` (in the script) nudges every *other* slide bubble outside the newly-active bubble's `footprintR` (its own radius plus however far its child ring blooms, computed at layout time), so the bloomed children always land in genuinely free space, never on top of a dimmed neighbor.
- **Dim AND clear space — this replaced an earlier "dim, don't reflow" rule.** The first version of this renderer deliberately left other bubbles in place (only dimmed) to avoid a second layout pass on every click; in practice that meant an expanding cluster visually sat on top of inactive neighbors, which read as cluttered rather than immersive. `relaxNeighbors()` is a one-shot relaxation (obstacle push + pairwise collision + a weak pull back toward each bubble's own resting spot, same family of technique as `packCircles()` in Step 3) computed once per click — not a live per-frame physics loop — so it's cheap enough to run on every click despite reintroducing a "reflow." Every other slide bubble still also gets the `dimmed` class; the two mechanisms (dim + clear space) work together, not instead of each other.
- **Motion is springy here on purpose.** `.slide-bubble`'s `left`/`top` transition uses `--spring-ease` (`cubic-bezier(.34,1.56,.64,1)`, a slight overshoot), not the calm `--ease` used everywhere else in this file's chrome/fades — bubbles nudging each other aside is supposed to read as organic and slightly wobbly, not a flat glide. Collapsing (`resetPositions()`) animates every bubble back to its packed resting position with the same spring curve. Don't reuse `--spring-ease` for anything that isn't a bubble physically displacing another bubble (dot zooms, opacity fades, etc. all keep the calm `--ease`/`ease`) — mixing the two curves is what keeps "organic bubble physics" and "calm UI chrome" reading as distinct, intentional layers rather than an inconsistent renderer.
- **Text children** (`.text-bubble`, one per `body[]` entry) show their full paragraph directly, already visible the moment the cluster un-hides — no further click needed, no second-level popup. See Step 3 for sizing.
- **Image children** (`.image-dot`, one per image) are small round thumbnails at rest. **Click one → it zooms in place** (`transform: scale()`, see Step 3) and a caption label fades in below it (a sibling text element, not inside any card). Click it again, or click elsewhere, to shrink it back. This zoom keeps the calm `--ease`, not `--spring-ease` — it's the image growing in its own reserved space, not one bubble displacing another.
- **Collapsing:** click the slide bubble again (toggles closed), click an expanded image dot again, or click empty canvas background — collapses whatever's open at that level and springs every displaced neighbor back to its resting position. A click anywhere inside an open `.childcluster` (a text bubble, the gap between children) does not bubble up and collapse the slide — reading text shouldn't accidentally close the cluster. Max one slide expanded at a time; expanding a new one auto-collapses whichever was open (springing its displaced neighbors back) before relaxing space around the new one.

## Step 5: Fill the template

Use `assets/bubble-template.html`. Copy it to your working directory.

1. Fill `{{BUBBLE_TITLE}}`, `{{BUBBLE_SUBTITLE}}`, `{{AUTHOR}}`.
2. Build the theme/slide/image structure from Steps 1–2 and replace the entire `{{BUBBLE_DATA}}` token (inside the `<script type="application/json" id="bubble-data">` tag) with one JSON object — no markdown fencing, no surrounding commentary:
   ```json
   {
     "themes": [
       {
         "name": "Theme Name",
         "slides": [
           {
             "index": 3,
             "heading": "...",
             "body": ["...", "..."],
             "images": [{ "src": "__IMAGE_SLIDE_3_1__", "caption": "..." }]
           }
         ]
       }
     ]
   }
   ```
3. **Image placeholders — critical, same rule as render-magazine:** never inline actual base64 data yourself. For every image on every slide, set its `src` to the exact literal token `__IMAGE_SLIDE_<slide>_<n>__` (`<slide>` = that slide's number, `<n>` = 1-based index of the image within that slide's `images[]`, so a slide with 2 images gets `__IMAGE_SLIDE_5_1__` and `__IMAGE_SLIDE_5_2__`). A code step after your output does a plain string substitution of each token for the real `data:` URI. Don't invent a filename or alter the token in any way.
4. `body` may be an array of multiple strings — each one becomes its own child bubble (see Step 3/4), so keep them as separate short entries; don't collapse them into one run-on paragraph.

## Step 6: Chrome

Copy the toolbar HTML/CSS/JS verbatim from the existing templates — do not rewrite it. 5-colour theme picker (blue, green, purple, orange, **red** — same palette as render-magazine, blue is the default). Dark mode toggle. Download PDF button targets `#canvas` (the full bubble map at its actual computed size), not `#canvas-wrap` (the scrollable viewport) or the whole page — targeting the wrapper would only capture whatever's currently scrolled into view.

**`.doc-header` (the title block) has a `::before` gradient scrim behind it — don't remove it or turn it into a solid card.** Unlike every other renderer's header, this one sits directly on top of a canvas that scrolls and packs bubbles edge-to-edge underneath it, so without something behind the text, whatever bubble happens to be scrolled under the title collides with it and both become hard to read — that's a real bug that was reported ("looks tacky"). The fix is a soft radial gradient in `var(--bg)` (the page's own background colour), strongest immediately behind the text and fading to fully transparent within its own bounds — deliberately NOT a flat, hard-edged rectangle, which would read as its own competing card fighting the bubbles for attention rather than the bubble map staying the main focus. If a future change wants the header more/less legible, adjust the gradient's stops/opacity, don't swap it for a solid background.

Scrollbars are hidden everywhere on every renderer (the global `* { scrollbar-width: none; ... } *::-webkit-scrollbar { display: none; }` rule right after the box-sizing reset) — scrolling still works, there's just no visible track/thumb. The canvas also supports click-and-drag panning since a big bubble map with invisible scrollbars needs an obvious way to explore it — don't remove that.

**`#canvas-wrap` must stay plain block layout — never `display: flex` + `align-items`/`justify-content: center`.** That combination on a scrollable (`overflow: auto`) container is a real trap: once `#canvas` is bigger than the wrap — the normal case for any deck past a handful of slides — flex's default ("unsafe") centering lets the child overflow equally on both sides, but scroll offsets can't go negative, so the overflow on the leading side (roughly the top-left of the map) becomes permanently clipped and unreachable by scrolling or dragging. This was a real bug: bubbles rendered visibly on screen but couldn't be scrolled to. Plain block layout has no such trap — `#canvas` sits at `(0,0)` inside the wrap and every pixel is reachable across the full scroll range. The "starts centered on load" look comes entirely from `layoutBubbles()` setting `wrap.scrollLeft`/`scrollTop` after layout — that's sufficient on its own; don't reach for flex centering to get it "for free."

**Click-and-drag panning must be able to start from ANYWHERE, including on top of a bubble — not just on bare canvas background.** An earlier version's `mousedown` handler bailed out immediately if the target was inside a `.bubble`, on the theory that bubbles need that mousedown for their own click-to-select. That worked when bubbles were small with visible gaps between them, but once bubbles got bigger and packed edge-to-edge (Step 3), there was often no bare background left to grab — nearly every mousedown landed on a bubble, panning couldn't start, and dragging across the map just fired each bubble's ordinary click in turn instead of panning through the space. The fix (see the template's `enableDragPan()`): always start tracking on `mousedown` regardless of target, then decide after the fact — via a `DRAG_THRESHOLD` (6px) of pointer movement — whether this was a drag (pan) or a click (select). If it was a drag, a `document`-level capture-phase `click` listener swallows the click event that would otherwise still fire on mouseup, so whatever the pointer started or ended on top of doesn't also react as if it'd been clicked. That listener is deliberately on `document`, not `#canvas-wrap` — a capture listener on a genuine ancestor of every possible click target is guaranteed to run before all of their listeners regardless of registration order; one on `#canvas-wrap` itself would only reliably beat bubbles (its descendants), not its own pre-existing background-click-to-collapse listener, since same-element listeners run in registration order. Don't reintroduce a target-based bail-out on `mousedown` — always track, decide by movement distance instead.

## Step 7: Validate before delivering

1. Grep for any remaining `{{` strings — anything left is a bug.
2. Confirm `{{BUBBLE_DATA}}` was replaced with valid JSON (e.g. pipe it through `node -e "JSON.parse(require('fs').readFileSync(0,'utf-8'))"`) — a syntax error here means the whole map fails to render.
3. Confirm no slide `heading`, `body` entry, or image `caption` contains the literal substring `</script` — that would terminate the data blob's script tag early and break the page. (Content this specific should never occur from a VLM caption or slide text, but check.)
4. Confirm every image across every slide has a correctly-numbered `__IMAGE_SLIDE_<slide>_<n>__` token and that no two images share a token.
5. Run `node --check` on the inline `<script>` block.

## Notes on extending

- If a third level of drill-down is ever needed (e.g. sub-bubbles on an image), follow the same pattern as the child cluster: fixed-size or content-sized circles, `packChildrenAroundParent()`-style random-seed-then-relax clustering around whichever single node is active, and — for anything that isn't an image — no dimension animation, just reveal already-positioned circles on un-hide. Don't introduce a rectangle at any level, don't lay children out on an even-angle ring, and don't introduce a different interaction model for one level.
- `packCircles()`, `packChildrenAroundParent()`, `seededRandom()`, and the tunable constants are the only things that should change if bubble density or organic "wobbliness" needs adjusting for a particular deck — don't add one-off inline overrides elsewhere in the script.
- Don't reintroduce a theme-level bubble, a ring-orbit layout baked into the initial map, a perfectly even-angle ring for child bubbles, a grid seed for the top-level pack, a rectangular content panel at any level (concentric or edge-anchored), or a legend — see the notes near the top of this file for why each of those was tried and rejected.
- Don't reintroduce per-theme colour or hue variation, literal hex colours unrelated to `var(--accent)`, or fixed (non-parent-relative) child sizing — colour is reserved for parent/child depth and size is reserved for content weight, both explained near the top of this file and in Step 3.
